# Handoff

Last updated: 2026-04-29

## Where we are

Smriti's purpose was reframed this session. It is no longer pitched as a
disk-cleanup tool (gdu owns that) — it is the **filesystem perception and
access tier** for agents and downstream tools. **kosha** owns document
comprehension on top.

Architecture sketch and next-steps notes are in:

- `docs/smriti-kosha-architecture-sketch.md` — the split, the seams, what each
  tier owns and deliberately does not.
- `docs/smriti-next-steps.md` — prioritized pre-design notes.

The README has been rewritten to match.

## What to pick up next

The priority code change is **scan event subscriptions** (item 2 in
`smriti-next-steps.md`). It is the load-bearing seam between smriti and kosha.
Without it kosha must poll or re-walk, defeating the architectural split.

Suggested first step: a dedicated **detailed design doc** for the event-stream
MCP tool. Open questions to answer:

- MCP-tool polling vs. SSE vs. Unix-socket pub/sub for v1 (recommendation:
  start with polling — `smriti_events_since(cursor, filters)`).
- Cursor lives in the consumer (kosha), not in smriti. Confirm this is right.
- Event schema versioning — how to evolve without breaking subscribers.
- Retention horizon: today `smriti prune` clears events > 30 days. Subscribers
  that fall behind for > 30 days lose events. Document or revisit.
- Coordinate with the planned `smriti watch` so they share an event shape.

The data is already in the `events` table — this is mostly an MCP surface +
filter + cursor design problem, not a schema rebuild.

## After that

In rough order:
1. CLI `smriti subscribe --since=… --types=…` for debugging.
2. `smriti watch` (inotify/fanotify) sharing the event shape.
3. systemd user service for `serve` + `watch`.
4. Decision on whether to retire the smriti-side dense embedding feature flag
   once kosha owns deep semantic search.
