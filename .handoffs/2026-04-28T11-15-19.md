## Context the next session needs

- `document_fts` is now **contentless FTS5** (`content=''`). It stores
  only the inverted index, not the source text. Two non-obvious
  consequences:
  1. SELECTs against `document_fts` columns return NULL — even on
     UNINDEXED columns (verified on SQLite 3.51, contradicts the docs).
     We link FTS rowid to `documents.rowid` and JOIN via rowid; do NOT
     try to add a `content_hash UNINDEXED` column expecting it to be
     readable.
  2. DELETE/UPDATE on FTS aren't supported without the original tokens.
     We don't delete during scan; orphan FTS rows are filtered out at
     query time by the `JOIN documents ON d.rowid = f.rowid`. If you
     ever need to compact, a full rebuild from `documents` is the path —
     but you'd need to re-read files (we don't store content anywhere).
- `enable_scan_pragmas` no longer disables `wal_autocheckpoint`. Don't
  re-add `wal_autocheckpoint=0` — it was the proximate cause of the WAL
  ballooning to 700MB+. The default 1000-page autocheckpoint is what
  we want.
- Insertion gating uses `INSERT OR IGNORE … RETURNING rowid`. If a doc
  with that content_hash already exists, no row is returned, and the
  FTS insert is skipped. If you change the documents schema, make sure
  `RETURNING rowid` still works.
