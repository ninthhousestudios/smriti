# handoff

## fixed: SIGBUS during concurrent scan + health

Root cause was **two writer-capable connections to the same DB**: `smriti-serve`
held a `db::open` (rw) connection for its lifetime, while `smriti scan` CLI
opened its own writer. Both processes shared the wal-index SHM mmap; SQLite
3.45.0 (bundled via rusqlite 0.31 / libsqlite3-sys 0.28) faulted in
`walCheckpoint` during the inline autocheckpoint hook.

Coredump 230071 (`smriti scan`) backtrace: `walCheckpoint →
sqlite3WalCheckpoint → sqlite3_wal_checkpoint_v2 → sqlite3WalDefaultHook →
sqlite3_step` from `Transaction::commit` in `flush_batch`. Faulting insn
`mov (%r12,%rdx,4),%edx` — wal-index hash table dereference on the SHM mmap.

The handoff hypotheses were all wrong: readers had `query_only=ON` so they
couldn't checkpoint; writer mmap was already 0; the SIGBUS was on the SHM,
not the main DB. The crashing process was the writer (scan), not the reader.

Fix:
- `src/daemon.rs`: smriti-serve runs migrations once with a writer, then drops
  it and uses `db::open_readonly` for the persistent connection.
- `src/mcp.rs`: `smriti_scan` MCP tool opens a fresh writer via
  `db::open(&config.db_path)` per invocation and drops it after.
- `src/db.rs`: removed the no-op `wal_autocheckpoint=0` from `open_readonly`
  (query_only readers don't checkpoint anyway) and updated the mmap_size
  comment to reflect the real reason it's disabled.

Verified: scan completes cleanly with 30 concurrent `smriti health` +
`scan-status` calls. No coredumps.

## fixed: smriti manifest including catalog dirs

Root cause: gitignore semantics don't expand `~`. Patterns like
`~/Downloads/`, `~/dev/`, `~/downloads/` in `~/.smritiignore` were treated as
literal patterns starting with `~` and never matched any path. Other catalog
patterns like `.local/`, `.cargo/`, `.bun/` work because they are
relative-anywhere directory names.

Fix in `src/ignore.rs::parse_smritiignore`: strip leading `~/` (and `!~/` for
negation) so the pattern becomes `/<rest>` — anchored to base_dir. For
`~/.smritiignore` (parsed with `base_dir=$HOME`) this gives the expected
behavior. Regression tests in `tests/ignore_test.rs`.

After fix: a fresh scan correctly catalogs `/home/josh/Downloads`,
`/home/josh/dev`, `/home/josh/downloads`; manifest output drops from 26813
entries under those paths to 0.

## context

- User smritiignore: `~/.smritiignore` (NOT `~/.config/smriti/.smritiignore`
  — the prior handoff was wrong; see `load_user_smritiignore` in
  `src/main.rs:174`).
- Hardened defaults: `src/ignore_defaults.txt` — catalogs node_modules/,
  target/, .cache/, etc.
- smriti-serve runs as systemd user service
  (`systemctl --user restart smriti-serve`).
- rusqlite is pinned to 0.31 (current is 0.39); SQLite 3.45.0 is bundled. An
  rusqlite upgrade is a separate task that may pull in unrelated WAL fixes
  but is not required for the SIGBUS fix.
